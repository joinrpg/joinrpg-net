# ADR009: Переход основной БД с Entity Framework 6 на EF Core

## Статус

Предложен, ожидает утверждения.

## Контекст

### Что есть сейчас

В проекте сосуществуют два поколения доступа к данным.

**Legacy (EF6, SQL Server).** Основная и самая большая часть данных живёт в
`JoinRpg.Dal.Impl` через `MyDbContext` — это `System.Data.Entity.DbContext`
(Entity Framework 6, Code-First). Он работает поверх **SQL Server**:
строка подключения `DefaultConnection` (`Data Source=...;Initial Catalog=joinrpg`),
а миграции применяются через `System.Data.SqlClient` (см.
`Joinrpg.Dal.Migrate/Ef6/JoinMigrationsConfig.cs`). Это сотни таблиц, ~130
навигационных свойств с `virtual` (ленивая загрузка), множество EF6-специфичных
вещей в `OnModelCreating` (`HasRequired`/`WithRequiredPrincipal`/
`WillCascadeOnDelete`/`IndexAnnotation`), кастомный логгер `EF6LoggerToMSExtLogging`,
`DbConfiguration`, `NullDatabaseInitializer`.

**Новое (EF Core, PostgreSQL).** Все DbContext-ы, добавленные за последние годы,
уже на EF Core поверх **PostgreSQL** (`UseNpgsql`):
`DataProtectionDbContext`, `JobScheduleDataDbContext`, `NotificationsDataDbContext`,
`IdPortalDbContext`. Они используют общий хелпер
`DbContextRegisterHelper.AddJoinEfCoreDbContext` и `LinqKit.Microsoft.EntityFrameworkCore`.

**Запуск миграций уже унифицирован.** `Joinrpg.Dal.Migrate` умеет прогонять
и EF6-миграции (`MigrateMyDbContextService`), и EF Core-миграции
(`RegisterMigrator<T>`) за один прогон через общий интерфейс `IMigratorService`
и `MigrationsLauncher`. То есть инфраструктура «применить все миграции при старте»
уже провайдеро-нейтральна.

### Почему это проблема

- **EF6 — тупиковая ветка.** Активно не развивается, не получает новых фич,
  плохо дружит с современным стеком .NET. Команда уже де-факто выбрала EF Core
  для всего нового.
- **Два ORM и два диалекта SQL** в одном приложении — двойная когнитивная
  нагрузка, разные API запросов (`System.Data.Entity.QueryableExtensions`
  против `Microsoft.EntityFrameworkCore`), разные правила маппинга, разные баги.
- **Нет современных возможностей** EF Core: `IAsyncEnumerable`-стриминг,
  compiled queries, `AsNoTracking` по умолчанию, `ExecuteUpdate`/`ExecuteDelete`,
  split queries, лучшая диагностика и трассировка.
- **Привязка к SQL Server.** Весь новый код на PostgreSQL, а самые большие данные —
  на отдельном SQL Server. Это два сервера БД в проде, две резервные копии,
  две точки отказа. Стратегическое направление — PostgreSQL.

### Две независимые оси миграции

Ключевое наблюдение: «переход на EF Core» — это на самом деле **две ортогональные
задачи**, которые важно не смешивать:

1. **Ось ORM:** EF6 → EF Core (на том же SQL Server).
2. **Ось СУБД:** SQL Server → PostgreSQL (перенос данных и схемы).

Их легко спутать, но риски и инструменты у них разные. Смешивать их в один
«большой взрыв» — самый опасный путь.

## Решение

### Принцип: декомпозиция и обратимость

1. **Сначала меняем ORM, СУБД оставляем прежней.** EF Core отлично работает с
   SQL Server (`Microsoft.EntityFrameworkCore.SqlServer`). Переводим `MyDbContext`
   на EF Core поверх того же SQL Server — это поведенчески нейтральное изменение,
   которое можно сравнить «один-в-один» со старым поведением.

   > Обратный порядок («сначала PostgreSQL, потом EF Core») практически невозможен:
   > поддерживаемого EF6-провайдера для PostgreSQL под современный .NET нет.
   > Пока мы на EF6, мы привязаны к SQL Server.

2. **Потом меняем СУБД.** Уже на EF Core переключаем провайдер на Npgsql и
   выполняем перенос данных SQL Server → PostgreSQL — это отдельный ADR и
   отдельный релиз.

3. **Максимум работы — до точки переключения.** Большую часть подготовки можно
   сделать, протестировать и выкатить в прод **на работающем EF6**, не меняя
   поведения. Это и есть главный приём снижения риска: к моменту собственно
   переключения ORM должно остаться как можно меньше неизвестного.

### Целевое состояние

- `MyDbContext` — `Microsoft.EntityFrameworkCore.DbContext`, регистрируется через
  тот же паттерн, что и остальные контексты (`AddJoinEfCoreDbContext`-подобный).
- Миграции `MyDbContext` — EF Core-миграции, применяются тем же
  `MigrationsLauncher`, что и остальные.
- Один ORM (EF Core), затем (в следующей фазе) один движок БД (PostgreSQL).

## Подготовительные задачи (выполняются и релизятся на EF6)

Список сознательно короткий. Сюда попадают только задачи, которые
**(а)** действительно снижают риск *именно* переключения ORM (а не переезда на
PostgreSQL — это фаза 2); **(б)** можно сделать, оттестировать и выкатить в прод
на работающем EF6 *с реальным выигрышем* (а не «то же самое, что и сам переезд,
только заранее»); **(в)** не меняют наблюдаемого поведения.

Несколько задач из ранних черновиков сюда **не** попали — почему, объяснено в
разделе [«Что осознанно не делаем заранее»](#что-осознанно-не-делаем-заранее).

### P0. Характеризационные тесты на реальной БД («golden master»)

**Что.** Достроить `JoinRpg.IntegrationTests`. Инфраструктура уже сильная:
`JoinApplicationFactory` поднимает **реальный SQL Server в Testcontainers**,
прогоняет настоящие EF6-миграции и строит полный веб-хост. Значит сравнение
«EF6 против EF Core» будет на одной и той же схеме, без моков.

Два уровня покрытия:
1. **HTTP-сценарии** (паттерн уже есть: `CreateEmptyProjectScenario`, XApi-тесты) —
   чёрный ящик controller → service → repo → DB. Самая надёжная гарантия, потому
   что тесты ORM-нейтральны по построению.
2. **Тесты репозиториев/сервисов через `factory.Services.CreateScope()`** —
   прямой вызов `IProjectRepository`/`IClaimsRepository` и проверка формы
   возвращаемого графа DTO/сущностей. Ловят различия трансляции запросов,
   которые HTTP-тесты могут замаскировать.

**Как проверять результат, не выписывая руками сотни ожидаемых значений.**
Подход golden master / approval: на EF6 сериализуем результат (граф DTO /
отрендеренный HTML) в утверждённый снапшот. После переключения те же тесты
гоняются на EF Core, и любой диф падает явно. Суть: нам не нужно знать
«правильный» ответ — нам нужно, чтобы EF6 и EF Core давали **идентичный** ответ
на одной и той же БД в контейнере. Это превращает вопрос «изменилось ли
поведение?» в механический диф.

**Что значит «достаточно».** Не гоняемся за процентом покрытия — целимся в
поверхность риска миграции: каждый метод репозитория с `Include`/проекцией/
`AsExpandable`, каждый путь записи, все многоуровневые `Include` и `.Select` по
навигациям. Дополнительный критерий полноты завязан на P1: гоняем набор, пока
счётчик ленивых загрузок не станет нулём (см. ниже) — любой ненулевой счётчик
означает непокрытое поведение, которое *изменится* на EF Core.

**Релиз.** Тесты зелёные на текущем EF6 + SQL Server до начала любых изменений.
Выкатывается первой.

### P1. Убрать зависимость от ленивой загрузки (lazy loading)

**Что.** В `JoinRpg.DataModel` ~130 `virtual` навигационных свойств — EF6
лениво подгружает их прокси. Сделать все загрузки явными (`Include`/`ThenInclude`
или отдельные запросы в репозиториях). Сюда же логично собрать вынос оставшихся
операций чтения из сервисного слоя внутрь репозиториев (правило проекта
«инкапсулировать чтение в репозиториях») — это одновременно убирает основные
точки ленивой загрузки.

**Зачем.** Самый коварный источник скрытых багов: код, «случайно» работавший на
ленивой загрузке EF6, на EF Core молча вернёт `null` или упадёт (ленивая загрузка
там требует отдельного пакета прокси, опт-ина и бросает после dispose контекста).
Явная загрузка одинаково работает и на EF6, и на EF Core.

**Как доказать, что поведение не изменилось.** Не на глаз, а измерением.
Эквивалентность гарантирована по построению: жадный `Include` возвращает ровно
те же строки, что подтянула бы ленивая загрузка, поэтому golden-master-снапшот
из P0 остаётся неизменным. Цикл такой: гоняем весь набор тестов со счётчиком;
каждое предупреждение — путь, зависящий от ленивой загрузки; добавляем `Include`,
убеждаемся, что предупреждение исчезло **и** снапшот идентичен. Когда счётчик —
ноль, отключение ленивой загрузки в момент переключения доказуемо является no-op.

**Существующая инфраструктура и её доработка.** В `EF6LoggerToMSExtLogging`
(`src/JoinRpg.Dal.Impl/EF6LoggerToMSExtLogging.cs`) уже есть детектор: эвристически
(один параметр с именем `EntityKeyValue1`) определяет вероятную ленивую загрузку,
инкрементит OTel-счётчик `joinrpg.mydbcontext.lazy_loads` и пишет warning с именем
таблицы из SQL. Сейчас он знает *таблицу*, но не знает, *кто* спровоцировал
загрузку — для поиска места это бесполезно. Предлагаемые доработки:
- **Ambient-контекст операции через `AsyncLocal<string>`**, выставляемый в
  базовом классе репозитория/сервиса (`DbServiceImplBase` или методах репозиториев)
  и читаемый в `LogCommand`. Дёшево, без связи с EF; даёт «ленивая загрузка из
  `ProjectRepository.LoadGroupWithTreeAsync`», а если течёт выше — имя сервиса.
  Это же значение повесить **тегом на счётчик**, чтобы телеметрия превратилась в
  приоритизированный список работ по компонентам.
- **Падение интеграционных тестов на ленивой загрузке.** В тест-хосте
  зарегистрировать хук/логгер, который бросает (или фиксирует и проверяет на
  teardown) при инкременте счётчика. Это превращает разовую чистку в постоянный
  регрессионный шлюз: новые ленивые загрузки добавить нельзя. Самая ценная
  доработка.
- Замечание: эвристика хрупкая (завязана на `EntityKeyValue1` — пропускает
  ленивую загрузку коллекций и может ложно срабатывать на легитимный `Find` по
  одному ключу). Реальная ценность не в парсинге SQL, а в `AsyncLocal` + падении
  тестов.

**Релиз.** Выкатывается по слоям/репозиториям, отдельными PR, на EF6.

### P2. Вынести маппинг в data annotations, где есть эквивалент

**Что.** Перенести из `OnModelCreating` на сами классы сущностей всё, что
выражается **атрибутами**: `[Required]`, `[Key]`, `[Column]`, `[ForeignKey]`,
`[InverseProperty]`, `[Index]`. Аннотации трактуются EF6 и EF Core **одинаково**,
поэтому такая конфигурация после переключения просто работает без переписывания
fluent-кода.

**Зачем.** Уменьшает объём самой рискованной части переключения — ручного
переписывания `OnModelCreating` с EF6-fluent (`HasRequired`/`WithMany`/…) на
EF Core-fluent. Каждая связь, ушедшая в аннотации, — это связь, которую не нужно
переписывать вручную.

**Граница.** То, у чего нет аннотационного эквивалента, остаётся fluent и
переписывается уже в момент переключения (см. P4b в разделе про саму миграцию):
`WillCascadeOnDelete(false)`, 1:1 по общему первичному ключу
(`WithRequiredPrincipal`: `ProjectDetails`, `UserAuthDetails`, `CommentText`,
`FinanceOperation` и т.п.), составные ключи, `IndexAnnotation` для уникальных
составных индексов.

**Релиз.** Чистый рефакторинг на EF6, поведение схемы не меняется
(проверяется P0 + сравнением сгенерированной схемы). Релизится по группам сущностей.

### P3. Контракт «первая EF Core-миграция пустая» (parity)

**Что.** В прототипе (отдельная ветка, без вывода в прод) собрать EF Core-модель
`MyDbContext` и инструмент сверки: первая EF Core-миграция должна быть **пустой**
(no-op `Up()/Down()`), а сверка модели с реальной схемой — зелёной. Зафиксировать
это автотестом «нет pending model changes».

**Зачем.** Гарантия, что EF Core не попытается пересоздать/переальтерить таблицы
боевой БД. Это «контракт перехода»: пока миграция не пустая — модель ещё не
соответствует схеме, переключаться нельзя.

**Релиз.** Прототип в прод не выводится; результат — план и тест, применяемые в
момент переключения. Ведётся параллельно с P2 (аннотации напрямую влияют на
совпадение модели и схемы).

## Что осознанно не делаем заранее

Чтобы список подготовки не раздувался мнимой работой, фиксируем, что **не**
является подготовкой к переключению ORM, и почему:

- **Абстрагирование `IUnitOfWork.GetDbSet<T>()` от типа EF6.** `GetDbSet<T>()`
  возвращает `System.Data.Entity.DbSet<T>` и используется по сервисному слою
  напрямую (`AccommodationInviteServiceImpl`, `AccommodationServiceImpl`,
  `KogdaIgraSyncService`). Но async-операторы (`Where`, `ToListAsync`, `FindAsync`,
  `Add`, `Remove`…) у EF6 и EF Core *исходно совместимы* — отличаются лишь
  `using`-неймспейсом и конкретным типом `DbSet<T>`. Поэтому смена типа возврата
  `GetDbSet<T>()` на EF Core-`DbSet<T>` перекомпилирует места вызова почти без
  правок: **это и есть переключение**, заранее тут делать нечего. Прятать `DbSet`
  за нейтральным `Add<T>/Remove<T>` имеет смысл только ради одновременной
  поддержки двух ORM — а это разовый односторонний переход, такой ROI низок.
  Полезную часть (вынос *чтения* в репозитории) мы и так делаем в P1.
- **Перевод запросов на Expression-предикаты / LinqKit ради миграции.** Это
  общее улучшение качества кода, ортогональное смене ORM. Более того:
  превращение ин-мемори `Func<>` в транслируемый `Expression<>` *увеличивает*
  риск на EF Core, т.к. его транслятор отличается от EF6 — выражение, которое EF6
  транслирует, может бросить на EF Core. Релевантная задача обратная — *проверить
  трансляцию существующих запросов* на EF Core, а это уже P0/P3.
- **Аудит DateTime/UTC.** SQL Server (и EF Core поверх него) терпимы к `DateTime`;
  строгость UTC — поведение Npgsql. Это пререквизит **фазы 2** (PostgreSQL), а не
  переключения ORM. Вынесено в фазу 2.
- **Аудит MARS / параллельных читателей.** `MultipleActiveResultSets=True`
  продолжает работать под EF Core *на SQL Server*; MARS ломается только на Npgsql.
  Значит это тоже пререквизит **фазы 2**, а не блокер переключения ORM.
- **Инвентаризация raw SQL / транзакций.** Проведена сейчас. По всему `src`
  единственные `FromSqlRaw`/`CreateExecutionStrategy` — в `JoinRpg.Dal.Notifications`,
  который **уже на EF Core**. В legacy-поверхности (`JoinRpg.Dal.Impl`,
  `JoinRpg.Services.Impl`, `JoinRpg.Integrations.KogdaIgra`) **нет**
  `BeginTransaction`, `SqlQuery`, `ExecuteSqlCommand`, `ObjectContext`,
  `TransactionScope` и командных интерсепторов (кроме логирующего
  `DatabaseLogFormatter`). Инвентаризировать нечего — отдельная задача не нужна.

## Сама миграция (фаза переключения ORM)

После подготовки переключение должно стать небольшим и обозримым:

1. Заменить пакет `EntityFramework` на `Microsoft.EntityFrameworkCore` +
   `Microsoft.EntityFrameworkCore.SqlServer` в `JoinRpg.Dal.Impl`.
2. Переписать `MyDbContext` на EF Core (`DbContextOptions`), включая **P4b** —
   остаток fluent-конфигурации без аннотационного эквивалента (каскады,
   1:1 по общему PK, составные ключи, уникальные составные индексы); зарегистрировать
   через общий хелпер.
3. Сменить тип возврата `IUnitOfWork.GetDbSet<T>()` на EF Core-`DbSet<T>` и
   реализовать `IUnitOfWork` поверх EF Core; места вызова в сервисах в основном
   перекомпилируются без правок (см. раздел «Что осознанно не делаем заранее»).
4. Заменить EF6-инфраструктуру: `MyDbConfiguration`/`[DbConfigurationType]`,
   `EF6LoggerToMSExtLogging`, `Database.Log`, `NullDatabaseInitializer` — на
   логирование/конфигурацию EF Core.
5. Сгенерировать **пустую** initial EF Core-миграцию (контракт из P3) и
   переключить применение миграций `MyDbContext` на `MigrationsLauncher`
   (EF6-ветку `MigrateMyDbContextService` удалить).
6. Отключить ленивую загрузку (после P1 это безопасно).
7. Прогнать всю сетку тестов (P0) — снапшоты должны совпасть.

## Фаза 2 (отдельный ADR): SQL Server → PostgreSQL

Только после стабилизации на EF Core + SQL Server. Пререквизиты, которые
осознанно не делались на этапе переключения ORM, потому что относятся к движку:

- **Аудит DateTime/UTC** — привести запись/чтение к консистентному UTC
  (Npgsql строг к `timestamp with time zone`).
- **Аудит MARS / параллельных читателей** — устранить опору на
  `MultipleActiveResultSets` (Npgsql его не поддерживает).

Собственно работа фазы:
- переключить провайдер на `Npgsql.EntityFrameworkCore.PostgreSQL`;
- перенести данные `joinrpg` из SQL Server в PostgreSQL (ETL/dump, сверка);
- пересоздать миграции под Postgres, решить вопросы enum (int vs native),
  идентити-колонок, регистронезависимости, коллаций;
- объединить инфраструктуру резервного копирования/мониторинга на один движок.

Это отдельное решение со своими рисками и здесь не детализируется.

## Последствия

**Плюсы:**
- Один ORM на весь проект; единые API запросов, маппинга, логирования, миграций.
- Доступ к возможностям EF Core (стриминг, `ExecuteUpdate/Delete`, split queries,
  `AsNoTracking`, лучшая диагностика).
- Снятие привязки к SQL Server открывает путь к единому PostgreSQL (фаза 2).
- Большая часть риска снимается заранее, на работающем проде.

**Минусы / издержки:**
- Подготовка (особенно P1) затрагивает много файлов.
- Тонкие различия поведения EF6 и EF Core (трекинг, материализация, дефолты
  каскадов, трансляция LINQ) требуют внимательного тестирования.
- Временно растёт связанность с тестовой инфраструктурой (P0 обязателен).

## Риски и их снижение

- **Скрытая опора на ленивую загрузку** → P1 (счётчик → ноль + падение тестов)
  поверх golden-master P0.
- **Несовпадение модели и боевой схемы** → контракт пустой миграции P3.
- **Различия трансляции LINQ EF6/EF Core** → проверяется характеризационными
  тестами P0 на реальной БД (не «исправляется» переписыванием на Expression —
  см. «Что осознанно не делаем заранее»).
- **Объём переписывания `OnModelCreating`** → заранее уменьшен выносом в
  аннотации (P2), остаток (P4b) — по документированной таблице соответствий.
- **Боль PostgreSQL (даты, MARS, диалект)** → вынесена в фазу 2.
